Evaluation & Improvement

Two things the brief implies but doesn't spell out: the extractor must be testable before deployment, and the system must improve on the documents this firm actually receives. Both come from measuring every stage and closing the loop.

TL;DR
  • Quality is measured on a fixed gold set before deploy (a CI gate) and improves on the firm's documents after deploy.
  • Metrics: field F1, schema-valid rate, hallucination rate, escalation rate, latency, cost per document.
  • Bootstrapped today with the public KIE-HVQA degraded-invoice set — before any client data exists.

What we measure

A handful of numbers, each tied to something the case cares about:

MetricWhy it matters here
Field accuracy / F1 (per field + overall)The core job — are the header + line-items right?
Schema-valid rateHow often output fits the contract first try (failure case ⓐ)
Hallucination rate on degraded scansThe legal risk — confident-but-wrong values (failure case ⓑ)
Escalation rateHow much traffic hits tier-2 / a human — drives cost & latency
Latency p50 / p95 & cost per docThe "real-world latency" and cost story, with real numbers

Before we deploy — the offline harness

We don't wait for production to find out if it works. We assemble a small gold set — labelled documents with known-correct answers — and run the whole pipeline against it, scoring the metrics above. The deploy only happens if it clears a bar, which makes evaluation a gate in CI/CD (see ⑦), not an afterthought.

We can start today, before the client gives us a single file: KIE-HVQA ships invoices with simulated degradation and reliability labels — a ready-made eval set for the scan / hallucination path — alongside the case's own sample invoice and a small hand-labelled seed.[1]

After we deploy — it improves on their documents

This is the part that makes it a living system rather than a one-off. The escalated and human-corrected cases the pipeline already captures[2] flow back into the gold set. Each release is re-scored, and the numbers are tracked over time in MLflow — so improvement (or regression) is visible, not anecdotal.

flowchart LR G[("📋 Gold set
labelled invoices")] --> R["▶️ Run the pipeline"] R --> M["📊 Score
accuracy · schema · hallucination · latency · cost"] M --> GATE{"meets the bar?"} GATE -->|yes| DEP["🚀 Deploy"] GATE -->|"no"| FIX["🔧 Tune thresholds · prompts"] FIX --> R DEP --> PROD["🏢 Real documents"] PROD --> ESC["🚩 Escalated + human-fixed cases"] ESC --> G

Left loop = before deploy (run → score → meets the bar? → tune until it does). Right loop = after deploy (real documents surface hard cases → humans fix them → they join the gold set → next release is measured against them). The firm's own document mix is what trains the system over time.

What the growing gold set unlocks

The labelled data we accumulate is the key that turns on the levers we deliberately left off at the start:

  • Tuned confidence thresholds — set the escalate-vs-accept cut-offs from data instead of guessing (the cascade thresholds are learned, exactly as FrugalGPT does it).[2]
  • DSPy prompt optimization — DSPy's optimizer compiles the extractor against an eval metric once we have examples; this is the headline DSPy feature we said was a later lever — eval is what unlocks it.[3]
  • Climbing the classifier ladder — labelled docs are what let us move from rules → trained classifier (see ④).
Why it matters: a proof-of-concept is a hypothesis, not a delivery. The evaluation harness turns every claim in this documentation — accuracy, hallucination-resistance, latency — into a measured number, and the feedback loop is the concrete answer to how the system improves once it is live.
