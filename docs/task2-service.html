One document = one invoice — and the splitter

The service assumes one PDF = one invoice. The case dataset breaks that assumption: it's a single PDF holding three unrelated invoices — Uber, WeWork, Cargo — one per page, and none have a usable text layer (one is an image, one a dark scan, one a corrupt/encoded text layer). Splitting is an ingestion concern, kept out of the extraction service: splitter.py bursts the file into one single-page PDF per page, and each becomes an independent job with its own retained document and audit trail.

Why split upstream instead of looping pages inside the service? It keeps the service contract clean (one job → one invoice → one result), gives every invoice its own audit trail and stored artifact, and mirrors where the work belongs on the platform — alongside format conversion and classification in ingestion.

The async API — why it's shaped this way

Extraction takes seconds, so a normal blocking request would hold the connection open and time out under load. Instead: POST returns a job id immediately (202), the work runs on a background worker, and the client polls a second endpoint for the result. User-facing latency becomes the time to accept the upload, not the time to extract.

sequenceDiagram participant C as Client / UI participant API as FastAPI participant S as Delta + Volume participant W as Worker participant FM as Model C->>API: POST /api/extract (PDF) API->>S: create job (PENDING) 📝 API-->>C: 202 {job_id} ⏎ instant API->>W: run in background W->>S: store raw PDF → Volume 📝 W->>FM: extract (text or vision) W->>S: SUCCEEDED + result 📝 C->>API: GET /api/jobs/{id} (poll) API-->>C: status + invoice C->>API: GET /api/jobs/{id}/audit API-->>C: full audit trail Note over S: 📝 every step is an append-only audit event

The client never waits — it gets a job id instantly and checks back later. Every status change (📝) is written to Delta, so progress survives a restart and the whole history is auditable; an in-memory dict would lose every in-flight job on redeploy.

The LLM layer — DSPy

One typed object is both the model interface and the validation contract. The Invoice Pydantic model is the single source of truth, shared by the signature and the validator — and tiering (fast default, escalate on failure) is a couple of lines. Field optionality is set by what real invoices carry: only total is required (an Uber receipt has no invoice number).

class LineItem(BaseModel):
    description: str; amount: float
    quantity: Optional[float] = None; unit_price: Optional[float] = None

class Invoice(BaseModel):              # the validation contract (case ⓐ)
    invoice_number: Optional[str] = None   # ride receipts (Uber) have none
    invoice_date: Optional[str] = None
    vendor: Optional[str] = None         # involved parties the brief calls out
    bill_to: Optional[str] = None
    currency: Optional[str] = None
    subtotal: Optional[float] = None; tax: Optional[float] = None
    total: float = Field(ge=0)         # the one amount every invoice has
    payment_method: Optional[str] = None
    line_items: list[LineItem] = []

class ExtractInvoice(dspy.Signature):  # text front-end
    document_text: str = dspy.InputField()
    invoice: Invoice = dspy.OutputField()
class ExtractInvoiceVision(dspy.Signature):  # image front-end (case ⓑ)
    page_image: dspy.Image = dspy.InputField()
    invoice: Invoice = dspy.OutputField()
# speed-first tiering: fast default, escalate only when needed
FAST    = dspy.LM("databricks/databricks-gemma-3-12b")               # tier-1 (vision)
PREMIUM = dspy.LM("databricks/databricks-meta-llama-3-3-70b-instruct") # tier-2

@retry(retry=retry_if_exception_type(TRANSIENT_ERRORS),  # ⓒ transient ONLY (not parse errors)
       stop=stop_after_attempt(3), wait=wait_exponential(min=1, max=10))
def _predict(predictor, **inputs) -> Invoice: return predictor(**inputs).invoice

def run_extraction(text=None, image_uri=None):
    predictor, inputs = (vision, image) if image_uri else (textmodel, text)
    try:
        return _predict(predictor, **inputs), TIER1
    except AdapterParseError:               # ⓐ won't fit schema → escalate to premium
        with dspy.context(lm=PREMIUM):           # premium also fails → raises → job FAILED (quarantined)
            return _predict(predictor, **inputs), TIER2

The three required failures, mapped: ⓐ bad output → DSPy re-asks, escalate to premium, then fail cleanly (job FAILED, quarantined); ⓑ image-only / corrupt-text scan → text-layer check → VLM; ⓒ flaky API → retry with backoff (parse errors are not retried). MLflow autolog traces every call for the model-audit layer.

SQL uses bound parameters, not string interpolation — correct for JSON with embedded newlines (an address line that broke a naive inline literal) and injection-safe. The JobStore Protocol is the swap point: at high poll volume, job status moves to a transactional store while Delta keeps the append-only audit + results. Anticipated questions

Q. Why split the multi-invoice PDF outside the service instead of extracting every page?
To keep the service contract honest — one document, one invoice, one result, one audit trail. A page loop would entangle "is this one invoice or several?" with extraction; on the platform that grouping decision belongs in ingestion next to classification. The splitter is ~30 lines and each split doc is independently retried, stored, and audited.

Q. The dataset's text layer looked fine — why vision?
Its text layer is corrupt/encoded (custom glyph encoding → garbled bytes), and the other pages are images. A printable-character ratio check detects "no usable text" and routes to the VLM — the same path a true scan takes (case ⓑ). This is more robust than trusting that any text layer is good.

Q. How is failure case (a) actually triggered if most fields are optional?
DSPy coerces the model output to the Invoice type and raises AdapterParseError on a mismatch — a missing total, a non-numeric amount, a negative total, or malformed JSON. We re-ask on the fast model, escalate to premium, and only then mark the job FAILED (quarantined) — never a silent bad row.

Q. Why retry only some errors?
Only transient errors (connection, timeout, rate-limit, 5xx) are retried with exponential backoff. A schema-parse error is deterministic — retrying the same prompt won't help, so it escalates the model tier instead. Retrying deterministic failures just burns latency and tokens.

Q. How does this scale to large document volumes?
The async contract means the API only does upload + enqueue; extraction is a worker concern that scales horizontally (in production, a queue + worker pool / Databricks Workflows). Delta holds results and the append-only audit; at high poll volume, job-status reads move to a transactional store behind the same JobStore interface — no service change.

Q. Where's the audit trail, and is it tamper-evident?
Two layers: a business-audit audit_events table (append-only, every state change) and a model-audit layer (MLflow autolog traces every model call). Both live in Unity Catalog with its access controls and lineage; Delta DESCRIBE HISTORY adds versioned who/what/when. The UI surfaces the trail live.

Q. Free Edition uses weaker models — is that a problem?
It's a config choice, not a code one. Free Edition disables Gemini/Claude, so the POC runs open-weight gemma-3-12b / llama-3-3-70b. They're imperfect on noisy receipts (e.g. an Uber total mis-read) — exactly what the premium tier-2 and human-review escalation exist for. A paid workspace swaps the tiers back with one env var; the pipeline is identical.
