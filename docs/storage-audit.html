Part 1 · Architecture

Storage & Audit Trail

The brief grades the data model on one thing: does it support an audit trail? The approach is to map each question an auditor will ask to a native Databricks primitive — most are configured, not built, which is a key reason for choosing Databricks.

TL;DR
  • Six audit questions, each answered by a native primitive: Delta history/time-travel, Unity Catalog lineage & access logs, MLflow tracing.
  • Plus one append-only event log (the only piece built) for the permanent record of state changes.
  • The audit trail is configured, not constructed — four of the six answers ship with the platform.

How the data is layered

We use the medallion pattern on Delta tables: Bronze = raw documents as received, Silver = validated, structured extractions, Gold = business-ready aggregates for reporting and totals. Unity Catalog governs all of it — which doubles as the confidentiality boundary a legal firm needs, so one client's documents never appear in another's view.

What an auditor asksDatabricks primitive that answers it
"Who changed this document's state, and when?"Our append-only event log — an immutable Delta table (the one piece we build)
"What did the data look like at time T?"Delta time-travel — query TIMESTAMP AS OF any past version[1]
"Who/what/when wrote this table?"Delta DESCRIBE HISTORY — version, userName, operation, timestamp per write[1]
"Where did this field come from?"Unity Catalog lineage — automatic, column-level, no instrumentation[2]
"Who accessed this client's data?"UC audit logs — the system.access.audit system table[3]
"How did the model produce this value?"MLflow tracing — exact input → prompt → output, per call[4]
Why Databricks for audit: the brief requires a data model that supports the audit trail. On most stacks, lineage, history, and access logging are custom builds. Here, four of the six answers above are platform features — the audit trail is configuration, not construction.

Limitations

Delta history isn't a forever-record by default. Table history retention defaults to ~30 days (logRetentionDuration), and UC audit logs default to 90 days (and are in Public Preview).[1] So the permanent legal record is our append-only event-log table — a Delta table we simply never delete from — with retention extended on the tables that need it. For row-level change capture we can also enable Delta Change Data Feed.[5]

And one design nuance: job status doesn't live in Delta

Delta is built for appends + history, not for a row updated constantly (a job flicking PENDING→RUNNING→DONE while it's polled). At POC volume we keep it in Delta for simplicity; at high poll volume, job status moves to a transactional store while Delta keeps the append-only audit and results. (Local dev writes Delta via delta-rs, no Spark; deployed uses Unity Catalog tables.)
