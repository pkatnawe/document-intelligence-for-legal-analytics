① Case Overview

Case Overview

A legal firm's Claims operation runs on paper — invoices, contracts, meeting minutes, emails — arriving as PDFs, Word, Excel, and scanned images. Today a person reads each one by hand. The goal is to turn that pile into structured, queryable, trustworthy data, so the firm spends its hours on judgment, not data entry.

Key takeaways
  • Problem: manual document handling costs an estimated $15–25 per invoice, all non-billable time.
  • Goal: ingest any format, extract the key entities (clients, amounts, dates, parties), and store them with a full audit trail — extensible to summarization/Q&A later.
  • Graded on: judgment and failure-handling — delivered as two architecture slides plus a PDF-invoice service and async API.

The real problem (why this matters to the firm)

This isn't "extract some fields." It's an operations problem with money attached. Manual invoice handling alone runs an estimated $15–25 per document[see Cost], and every hour a lawyer or paralegal spends filing and re-keying is a non-billable hour. The documents are messy and inconsistent, they have to be findable later, and — because this is legal — every step has to be defensible: you must be able to show how a number was produced and who touched it.

The system augments the team rather than replacing it: people apply judgment; the system handles the mechanical reading at speed and scale. It extracts what it can, flags what it is unsure about for human review, and leaves a complete audit trail behind.

What we're graded on — three deliverables

#DeliverableScope
1Two slides — the architecture, and the execution pipelineAll document types (breadth)
2A Python service — extract header + line-items via LLM/VLM, handling bad output, image-only scans, and flaky API callsPDF invoices only (depth)
3An async API — submit a PDF, get a job id instantly, poll for the resultPDF invoices

Part 1 is a wide design exercise; Part 2 is a narrow build. We keep them apart so the platform vision doesn't bloat the code.

The real constraints (the project as a whole)

Everything we design has to answer these — they come straight from the brief, and they're the lens for every decision in this site:

ConstraintWhat it really demands
Diverse documentsIngest + classify many types and formats (PDF/Word/Excel/image), including scans with no text layer
Failure at every stageNever a silent wrong answer — escalate when unsure, fall back to a human, record everything
Scale to large volumesThroughput grows without a rewrite as the firm sends more
Audit trailA data model that proves who/what/when, and how each value was produced
Downstream useResults feed reporting, search, summarization, and aggregation
ExtensibilityRoom to add LLM summarization / Q&A later without re-architecting
Confidentiality impliedLegal data is sensitive — governance over who sees what, and what may go to a model
The rubric, in their words: "We are not looking for a production-complete implementation. We are looking for the logic and judgment in how you structure the service and handle cases where things go wrong." Clean structure and graceful failure beat feature count — every decision here is measured against this.

Why Databricks

The case doesn't require Databricks — it's our choice, because the constraints above map onto things it gives us natively: Apps host the API, Foundation Model APIs do the LLM/VLM work, Delta + Unity Catalog handle storage, audit, and confidentiality, and Workflows handle scale. The audit trail especially is configuration, not construction (see ⑤).

Deployment note: the proof-of-concept runs on Databricks Free Edition — serverless-only, outbound internet limited to trusted domains, non-commercial.[1] Production moves to a paid workspace using the same code and the same deployment bundle.
