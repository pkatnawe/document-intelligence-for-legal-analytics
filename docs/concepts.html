Concepts & Glossary

Reference for every concept behind the design, the research, and the solution — defined plainly, with why it matters for this case and a one-line summary. Start here if a term anywhere in the documentation is unfamiliar.

Design concepts (our solution)

The handful of ideas the whole architecture is built on.

Confidence-gated escalation (a "cascade")

At each step, try the cheap/deterministic path first; if it isn't confident, escalate to a stronger model; if that's still unsure, hand to a human. Why for this case: it gives all three things at once — graceful failure (never a silent wrong answer), low cost, and low latency. It's the spine of both slides, and it's the published cascade idea from FrugalGPT.

In one line: "Cheap path for the easy majority, escalate only when the model isn't sure, human as the last resort."

Model tiering

The escalation in concrete terms: a fast default model (tier-1) handles most documents; only low-confidence ones reach the premium model (tier-2). The deployed POC runs gemma-3-12b → meta-llama-3-3-70b-instruct (open-weight, what Free Edition serves); production swaps to gemini-3-5-flash → claude-opus-4-8. Why: one dial that serves both speed and quality — and with DSPy, swapping a tier is a one-line change.

Classification-guided extraction

Figure out what the document is first, then extract with logic tailored to that type — rather than asking one model to do both at once. Why: it's more accurate and needs far less labelled data (a 2026 Nature paper shows ~18 F1 points better, zero-shot), and it lets the cheap classifier filter out non-invoices before the expensive extractor runs.

The input guard

Because the classifier runs first, "is this actually an invoice?" is a free check: a contract sent to the invoice extractor is rejected with a reason instead of having invoice fields hallucinated onto it. Why: a failure-handling win that costs nothing extra.

Graceful degradation

The system is never allowed to be confidently wrong. Anything it can't handle with confidence is escalated, and ultimately flagged for a person — every such decision recorded. Why: for a legal client, a flagged uncertainty is safe; a silent fabrication is not.

Async "accept-and-poll" (HTTP 202)

The API takes the file, returns a job id immediately, does the slow extraction on a background worker, and lets the client poll for the result. Why: extraction takes seconds, so this keeps user-facing latency tiny and survives load — exactly what Task 3 means by "real-world latency."

The audit trail, in layers

Different audit questions need different evidence: our append-only event log (state changes), Delta history/time-travel (data history), Unity Catalog lineage (where a field came from), UC access logs (who looked), and MLflow (how the model produced a value). Why: the case grades the data model on the audit trail, and four of those five are Databricks-native — see ⑤.

RAG (retrieval-augmented generation)

For future Q&A/summarization: retrieve the most relevant passages (by meaning, via embeddings), then ask an LLM to answer over those passages with citations. Why: it's how the "extensibility" requirement gets satisfied later without a re-architecture — see ⑥.

Pipeline / ML

TF-IDF

Turns text into numbers by scoring each word by how often it appears here × how rare it is everywhere. Distinctive words ("remittance") score high, "the" scores ~0 — giving a cheap classifier something to work with. In our plan: rung 2 of the classification ladder.

In one line: "TF-IDF weights words by how distinctive they are, so a lightweight model separates invoices from contracts without an LLM."

kNN (k-Nearest Neighbors)

Classify a new document by finding the most similar already-labeled ones and taking a majority vote — no training, just comparison in vector space. In our plan: a rung-3 option, riding on embeddings.

In one line: "kNN labels a doc by looking at the most similar labeled docs and voting."

Embeddings

A model that turns text into a vector where similar meaning lands nearby — so "invoice" and "bill" are close even though the words differ. In our plan: the basis for future semantic search/Q&A; endpoints bge-large-en, gte-large-en.

In one line: "Embeddings map text to vectors where closeness means similar meaning — that's what powers semantic search."

VLM & OCR

OCR reads characters off an image (deterministic, fails by garbling characters). A VLM is an LLM that can see images and reason over layout (higher accuracy, but can hallucinate). In our plan: OCR reads scans for classification; the VLM extracts scans — in the POC that's gemma-3-12b (vision), with production escalating to the vision-capable claude-opus-4-8.

Framework / tooling

DSPy

Program an LLM with a typed signature — declare the inputs and a Pydantic output schema — instead of hand-written prompt strings; DSPy coerces the reply into that type. In our plan: it's the LLM interface and the validation contract in one object, which is exactly the separation the brief asks for. Under the hood it coerces output via a Pydantic TypeAdapter; on a bad parse it raises its own AdapterParseError (not raw pydantic.ValidationError) — that's the catch point for failure case ⓐ.[1]

In one line: "DSPy makes the model's output a typed contract, so validation is built into the interface rather than bolted on."

MLflow tracing

Auto-records every LLM call — input, prompt, model+version, output, tokens, latency — as a durable trace. In our plan: the model-audit layer (paired with the business event log), so each extraction is reproducible. DSPy emits these automatically.

In one line: "MLflow tracing captures every model call end-to-end, so each extraction is reproducible — critical for legal."

Pydantic

Defines data shapes as typed classes and validates against them, raising a clear error on bad data. In our plan: the Invoice schema is a Pydantic model — the single source of truth shared by the DSPy signature and the validator. It is "the expected schema."

Platform / ops

CI/CD

CI runs your tests automatically on every push, so broken code is caught immediately. CD auto-deploys what passes — for us, databricks bundle deploy stands up the app. In our plan: the "platform maturity" line on Slide A — repeatable, auditable releases; bad code never reaches the workspace.

In one line: "CI tests every push; CD auto-deploys what passes via a Databricks Asset Bundle — releases stay reproducible and safe."

Medallion (Bronze / Silver / Gold)

A layering convention: Bronze = raw, Silver = cleaned/validated, Gold = business-ready aggregates — each a set of Delta tables. In our plan: maps onto the pipeline and gives the audit story via Delta time-travel.

Delta · Unity Catalog · Vector Search

Delta = the table format (transactions + time-travel history). Unity Catalog = governance (permissions, lineage). Vector Search = an index over embeddings for fast semantic lookup. In our plan: audit, legal confidentiality, and future Q&A respectively.

Databricks Asset Bundle (DAB)

Infrastructure-as-code for Databricks (recently renamed Declarative Automation Bundle): one version-controlled databricks.yml declares the resources — our App, its secret, and per-environment targets (dev/prod) — and the CLI reconciles the workspace to it. In our plan: the same bundle promotes the Free-Edition POC to a paid production workspace by changing one flag (-t prod); deploys are reproducible, reviewable in Git, and (in prod mode) run as a service principal.

In one line: "The whole deployment is one YAML file — promote dev→prod with a flag, no clicking, fully auditable."

Databricks Apps · Foundation Model APIs · serving endpoint

Databricks Apps host our FastAPI service on serverless compute, behind workspace OAuth. Foundation Model APIs expose hosted LLMs/VLMs as OpenAI-compatible serving endpoints (pay-per-token, or provisioned throughput at scale). Service principal = the app's machine identity; secret scope = where the token lives (never in code). In our plan: the app, the models, the storage, and the audit all sit inside one governed platform.

Case & domain terms (from the brief)

The vocabulary the case itself uses, in plain language.

Claims line of business

The insurance/legal function the POC targets — processing the paperwork tied to claims (invoices, contracts, minutes, emails). Why it matters: it's a high-volume, document-heavy area where automation pays off fastest.

Entity extraction · metadata

Entities are the key facts the brief names — client names, invoice amounts, dates, involved parties. Metadata is data about the document (type, source, dates) used to find and route it. In our plan: the Invoice schema's fields are the entities; storing them makes documents queryable.

Involved parties

Who a document is between — for an invoice, the vendor (who issued it) and the bill-to (the client). Why it matters: linking documents to parties is how a claim is reconstructed across invoices, contracts, and emails.

Audit trail · downstream use

Audit trail = a defensible record of who/what/when and how each value was produced (required because it's legal). Downstream use = what the stored data feeds: reporting, search, summarization, aggregation. In our plan: the append-only event log + Delta history answer the audit ask; Gold tables + the vector index serve the downstream ask.

Research grounding — why we took these ideas

Three papers underpin the design. We read them for the main idea only — this is a case study, not a reproduction — and each one maps onto a choice we'd already reasoned our way to.

FrugalGPT — the cascade Stanford, 2023

Main idea: query a cheap model first, score its answer with a tiny scorer, and only escalate to a pricier model if the score is below a learned threshold. A 3-model cascade matched GPT-4 at up to ~98% lower cost.

What we took & why: this is our confidence-gated escalation (tier-1 fast → escalate to tier-2). It's the published basis for "fast and cheap on the easy majority, premium only on the hard tail" — our cost and latency story. (Bonus: one of their test sets, OVERRULING, is legal text.)[1]

Seeing is Believing? (KIE-HVQA) — the failure mode ByteDance, NeurIPS 2025

Main idea: under visual degradation (blur, occlusion), vision-language models lean on language priors and hallucinate plausible-but-wrong document values instead of admitting they can't read it. They build a benchmark (KIE-HVQA) and train a model — via reinforcement learning with a reward for refusing when unsure — that beats GPT-4o by ~28% on hallucination-free accuracy under degradation.

What we took & why: the evidence for two of our calls — use deterministic OCR (not a VLM) to classify scans, and route low-confidence scans to escalation/human rather than letting a model guess. Signal uncertainty, don't fabricate.[2]

📦 Usable data: KIE-HVQA is public on HuggingFace and includes invoices with simulated degradation — an off-the-shelf eval set for our scan / hallucination handling. huggingface.co/datasets/bytedance-research/KIE-HVQA

Classification-guided LVLM — the architecture Nature Sci. Reports, 2026

Main idea: for mixed document types, decouple type-classification from extraction — classify first (they even offer a training-free keyword classifier over OCR text), inject a type-specific prompt, then run the vision-language model. Zero-shot, it beat a supervised baseline by ~18 F1 points.

What we took & why: the whole shape of our pipeline — classify → route → type-specific extract — and our keyword-over-OCR classifier (decision C) is literally their training-free option. Peer-reviewed proof that our architecture and our cheap classifier are the accurate, low-supervision design.[3]

📦 Usable code: their Multi-VIE code and a fine-tuned Qwen2.5-VL model are public. github.com/FairmeHIT/Multi-VIE

Models we could evaluate

Keep these as benchmark candidates — we can A/B several later against the same invoices (the KIE-HVQA degraded set is ready-made for it).

Running now (Free-Edition POC)

The deployed app actually runs open-weight tiers, because Free Edition disables the proprietary models: tier-1 databricks-gemma-3-12b (vision-capable — reads the scanned invoices) and tier-2 databricks-meta-llama-3-3-70b-instruct (text). These are what produced the verified extractions.

Production target (paid workspace)

Swap the two env vars and the same code runs on databricks-gemini-3-5-flash (tier-1, fast + vision) escalating to databricks-claude-opus-4-8 (tier-2, "complex extraction… document understanding", and vision-capable so the escalation also covers scans). This is the whole point of model choice being config, not code.

Specialist candidate — Qwen2.5-VL-7B-SFT (Multi-VIE)

The Nature team's vision model, fine-tuned for exactly this task (multi-type visual information extraction) and public on HuggingFace. A strong thing to evaluate for the extractor / tier-2 slot — after fine-tuning it reached 93.65% F1 on their dataset, and showed robustness to seals, watermarks, and low contrast. model

⚠ Caveat: serving a custom GPU model needs a paid Databricks workspace — Free Edition is serverless with no GPU / custom-model serving. So it's a production-tier candidate (or for offline eval), not something we stand up on Free Edition for the POC. Fits the "POC on free, production on paid — same code, same bundle" framing.
