Execution Pipeline

The step-by-step slide: how a document moves through the system. The idea running through all of it is confidence-gated escalation — the cheap, deterministic path runs first, and the expensive LLM is used only when the cheap path is not sure.

TL;DR
  • Flow: read (any format) → classify → route by type → extract → validate → store.
  • The three required failure cases are handled: unparseable output, image-only scans, transient API errors.
  • Scanned / image-only documents are a first-class path — classified by type, then read by a classification-guided vision model (no OCR — it preserves layout).
  • Every decision is recorded to the audit trail.

Read it (top). Every input type — PDF, Word, Excel, image — meets one question: is there a text layer? Digital files are read directly; image-only files (scans, photos) are rendered to a page image — no OCR, because OCR flattens the spatial layout invoices depend on. Classify: predict the document type (rules over text for digital, a lightweight image classifier for scans) — confident → route, unsure → escalate to an LLM/VLM classifier, genuine unknown → human. Route & extract — classification-guided: the type selects a type-specific prompt + schema, and the vision model reads the page image directly (digital docs use the text LLM); decoupling classify from extract is the published state of the art for mixed documents — ~18 F1 over a baseline.[2] Each escalates to a stronger model only on a failed check.[1] Every 📝 step is recorded.

Why that one pattern earns its keep

Confidence-gated escalation isn't just tidy — it solves three problems at once:

🛟 Graceful failure

A doc the cheap path can't handle is escalated, not guessed at. Worst case it's flagged for a human — never a confident wrong answer. Essential for legal.

💰 Lower cost

The expensive model only runs on the small fraction below the threshold. The cheap path absorbs the easy majority.

🎓 Free training data

Every escalation and human correction is captured — the labels that feed the evaluation loop and teach the cheap path to handle more over time.

A PDF either has selectable text (read it) or it's a scan (send the image to the vision model — case ⓑ). The result clears two gates: it must fit the schema (ⓐ — re-ask, then escalate), and its line items must reconcile with the total (a correctness check, since a weak model can return a well-typed but wrong number). A failure at either gate escalates to a stronger model; if that still doesn't resolve, the document is flagged for a human — the final step. Flaky API calls retry with back-off (ⓒ). Every outcome — saved or flagged — is 📝 recorded.

The final step is always a human. Nothing the system can't handle with confidence is dropped or guessed — it's escalated to a stronger model and, failing that, routed to a person, with every decision on the audit trail. For a legal client, a flagged uncertainty is safe; a silent fabrication is not.
