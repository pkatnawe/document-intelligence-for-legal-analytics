Dev / Test / Deploy

Testing the models does not require deploying to Databricks first — which shapes the development workflow: build locally against the live endpoints, deploy last.

You test from your laptop. Databricks serving endpoints are ordinary HTTPS endpoints, and the VS Code extension already sets up auth. Local Python calls the live models — no deploy needed. Deploying the app is only the final packaging step. So the loop is: write locally → call live models → deploy last.

Two phases, same code. In dev, code runs on your laptop and reaches the model over HTTPS using your extension profile. When deployed, the same code runs in a Databricks App and reaches the same endpoint over the internal network, authed by the app's service principal. Nothing about the code changes.

The smoke test (our first real action)

Sixty seconds to prove the whole premise — that we can call a live model with a token:

databricks-sdk
curl
# pip install databricks-sdk openai
from databricks.sdk import WorkspaceClient
w = WorkspaceClient()                 # uses the extension's profile
client = w.serving_endpoints.get_open_ai_client()
resp = client.chat.completions.create(
    model="databricks-gemini-3-5-flash",
    messages=[{"role":"user","content":"Reply with: ok"}],
)
print(resp.choices[0].message.content)
curl -s "$DATABRICKS_HOST/serving-endpoints/databricks-gemini-3-5-flash/invocations" \
  -H "Authorization: Bearer $DATABRICKS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"messages":[{"role":"user","content":"Reply with: ok"}]}'
Already verified (2026-06-08): auth returned HTTP 200 as pkatnawe@uwo.ca; the serving-endpoints and Apps and Jobs APIs all returned 200 (there's even one app already in the workspace). So the token works, the models are reachable, and the deploy target is real. The token now lives in a Databricks secret scope (injected via valueFrom) — never written to the repo.
End-to-end verified live (2026-06-09): the service ran against the case's own dataset on Free Edition with the Delta store in Unity Catalog. The invoice's text layer was corrupt, so it routed to the vision path; gemma-3-12b extracted it (invoice 2011981, Cargo Collective, $99 USD); the job + result + audit trail persisted to workspace.invoice_poc Delta tables, and DESCRIBE HISTORY showed the versioned who/what/when audit primitive.
Free Edition model reality (important): the proprietary models (Gemini, Claude) are disabled on Free Edition — calls return PERMISSION_DENIED: rate limit of 0. Open-weight models do serve, so the POC uses gemma-3-12b (vision) and meta-llama-3-3-70b-instruct / gpt-oss-120b (text). Production on a paid workspace swaps the tiers back to Gemini/Claude — same code, one env change. This is the concrete reason model choice is config, not code.

What's actually callable

How it productionizes — CI/CD

Not required by the case, but a one-line mention signals operational maturity. In plain terms: CI runs your tests automatically on every push; CD auto-deploys what passes. (Full definition on ★ Concepts.)

flowchart LR DEV["✏️ Write code"] --> PUSH["⬆️ Push to GitHub"] --> CI{"Tests pass?"} CI -->|no| STOP["🛑 Blocked — never ships"] CI -->|yes| CD["🚀 Auto-deploy"] --> APP["✅ Live in Databricks"]

The test gate means broken code never reaches the workspace, and the deploy itself is one command because it's a Databricks Asset Bundle (below).

Deployment is declarative — the Asset Bundle

databricks.yml is a Databricks Asset Bundle (recently renamed a Declarative Automation Bundle) — infrastructure-as-code for Databricks. Rather than clicking through the UI, the whole deployable unit is declared in one version-controlled file: the App, the secret that injects the token, and per-environment targets. The CLI reconciles the workspace to that declaration.

📦 One unit, three commands

bundle validate checks config, bundle deploy creates the App + secret, bundle run starts it. The whole service is one declarative artifact, not a pile of manual steps.

🔁 Promote dev→prod by a flag

The same bundle deploys to the Free-Edition POC (-t dev) or a paid production workspace (-t prod) — only workspace.host changes. Same code, no rewrite. Prod mode also runs the app as a service principal and can gate on the Git branch.

🔐 Secrets by reference

The PAT lives in a Databricks secret scope; databricks.yml declares it as a resource and app.yaml maps it in via valueFrom. The value is resolved at runtime — never committed to Git.

Why it's a strong answer: declarative means the deploy is reproducible (re-running converges, doesn't double-create), reviewable (every infra change is a Git commit — who/what/when), and reversible. That's also part of the audit story: infrastructure changes are version-controlled and attributable, and production runs under a managed identity, not a personal token.

Deployed & verified (2026-06-09): the bundle deployed the FastAPI service to Databricks Apps; uploading the 3-invoice case dataset through the live app split it into three single-invoice jobs (Uber, WeWork, Cargo), each extracted on a Databricks-hosted vision model and persisted to Delta + the UC Volume with a full audit trail — and every model call is captured in the MLflow experiment.
