A run-of-show for presenting to experienced engineers: trade-offs, data flow, and bottlenecks — not marketing. Skip the high-level fluff; they value design viability. Present from ▣ The Two Slides and the live app; keep this open as the map.

The one idea everything hangs on — say it in the first minute, then let it recur. Confidence-gated escalation: a cheap, deterministic path handles the easy majority; escalate to a stronger model only when unsure; never guess — the final step is always a human. It shows up at classify, at extract, at validate, in the cost math, and in the improvement loop. If they remember one sentence, it's this.

③ AI lifecycle component (minute 2–6)

  • Model type — VLM/LLM, not XGBoost. This is open-vocabulary structured extraction from arbitrary layouts and images, not fixed-feature tabular prediction — there are no engineered features to boost over. A VLM reads layout directly; a classical model can't ingest a scanned receipt.
  • Why a cascade (FrugalGPT), not one big model. Fast model on the easy bulk, premium only on the hard tail — the cost + latency story, backed by the paper (≈98% cheaper to match a top model).
  • Training vs inference sync. We don't train a model — inference-only against hosted FMs, so there's no train/serve skew. What we "train" is the prompt: GEPA compiles the DSPy signature against an eval metric, later, from accumulated examples.
  • Vector DB / RAG — deferred, designed-for. Future Q&A is RAG: page-chunk embeddings (bge/gte) in Databricks Vector Search, tagged with client/date/party for filtered retrieval. Not built in the POC, but the storage choices today make it an add-on, not a rewrite.

Acknowledge the technical debt (name it before they do)

  1. Open-weight accuracy on Free Edition. gemma-3-12b misreads dense receipts (the Uber total). Mitigation in place: reconciliation catches it and flags for review; production swaps tier-2 to vision-capable claude-opus-4-8. This is the weakest point — I'd lead with it.
  2. Vision escalation is a no-op on Free Edition — tier-2 (llama-3-3-70b) is text-only, so a scan can't be re-read; it goes straight to human. Real escalation needs the paid workspace's vision premium model.
  3. Job-status in Delta won't survive high poll volume (hot-row updates). Move status to a transactional store (e.g. Lakebase) behind the same JobStore interface; Delta keeps results + audit.
  4. Thresholds are hand-tuned (text-layer ratio 0.6, reconcile tolerance, confidence gate) — guesses until we have data; GEPA + the eval loop tune them on the firm's real invoices.
  5. Reconciliation only catches arithmetic inconsistency — a wrong-but-consistent value (e.g. a one-character invoice-number OCR slip) isn't caught; that's what the premium model + human review are for.
  6. Single-type in code (invoice). Multi-format ingestion + classification are platform-design only, not built.
  7. Confidentiality/PII not implemented — redaction-before-model and model-placement (in-perimeter vs cross-geo) are flagged open, for the firm's experts.

The improvement story (if asked "how does it get better?")

The failures we log are the training signal. Human-flagged cases (schema fails, reconcile mismatches, corrections) accumulate into a gold set, then GEPA (Genetic-Pareto — the current DSPy optimizer, ICLR 2026) evolves the prompt by reflecting on that rich textual feedback, not just a scalar score — from as few as ~10 examples (vs MIPROv2's 200+). Same loop tunes every threshold. So it improves on this firm's documents, measured in MLflow, not anecdotally.

Q&A buffer — likely questions, crisp answers

Why DSPy over plain prompting / LangChain? Typed signature = interface + validation in one; one-line model/tier swap; GEPA optimization path. Less glue than LangChain.

Why not fine-tune a model? Optimizing the text layer is a legitimate update mechanism and the right first one here: sample-efficient in the low-data regime (GEPA from ~10 cases vs thousands), auditable and reversible (a diffable prompt edit, not an opaque weight delta — it matters for a legal client), and inference-only avoids train/serve skew + GPU cost. Distil into weights only once a behaviour is stable and proven on the firm's data. (Lee, Take Text Optimization More Seriously, 2026.) A fine-tuned Qwen2.5-VL stays a later production-tier candidate.

How do you know an extraction is right? Two gates — schema (well-typed) and arithmetic reconciliation (well-formed). Neither proves semantic truth, so uncertain results escalate then go to a human; correctness is measured on the gold set.

How does it scale? Async decouples API latency from extraction; workers fan out via Workflows; endpoints move to provisioned throughput; job-status moves to a transactional store. No rewrite — levers per stage.

Where's the audit trail? Append-only audit_events + MLflow traces + Delta history + UC lineage/access — mostly configured, not built. That's the main reason for Databricks.

What breaks first under load? Job-status polling against Delta. Fix is the transactional store; everything else (workers, endpoints) scales horizontally.
