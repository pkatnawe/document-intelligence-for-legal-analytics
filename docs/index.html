Overview

Document Intelligence for Legal Analytics 🔒 design locked · v1 · 2026-06-08

An automated pipeline that turns a legal firm's mixed Claims documents — invoices, contracts, minutes, emails — into structured, queryable, auditable data.

Executive summary
  • Outcome: extract the key data from any document format at ~$0.005 per document, versus $15–25 handled by hand.
  • How: a cheap, deterministic path handles ~90% of documents; a premium model and human review handle the difficult remainder.
  • Trust: nothing is ever silently wrong — uncertain cases escalate, then reach a person, and every step is recorded to a Databricks-native audit trail.
  • Improves: accuracy rises on the firm's own documents over time, measured on a fixed evaluation set.
▣ The deliverable: the two slides are on The Two Slides (presentation-ready). The remaining pages are the supporting documentation.

The work has two parts, kept deliberately separate so the platform design does not inflate the build:

Part 1 — the architecture

Two slides covering the whole platform across every document type — failure handling, scale, and audit. Scope: breadth.

Part 2 — the service built · deployed

A focused Python service that extracts header + line-items from PDF invoices, behind an async API with an upload UI. Running live on Databricks Apps, persisting to Unity Catalog. Scope: depth. See ⑨ Service.

Breadth on the slides, depth in the code — the rubric rewards judgment over feature count.

Where the design stands

✅ Known knowns (locked)

Classify-then-route by document type · rules-over-text classifier with confidence-gated escalation · DSPy typed extraction · speed-first model tiering (Gemini Flash → Claude Opus) · graceful failure (escalate → human) · Databricks-native audit (Delta history + UC lineage/access + MLflow + append-only log) · medallion storage · async API · eval-driven before & after deploy · POC on Free Edition, production on paid. Full log on ⑧ Decisions.

❓ Open questions (by design)

Sensitive data: what may be sent to which model — for the firm's experts; not built (note: tier-1 Gemini is cross-geo, tier-2 Opus is in-perimeter). Document mix: which classifier the type-set needs (training-free vs trained)? Thresholds: tune from real data. All three resolve by testing on the firm's actual documents — see Evaluation & ⑧.
