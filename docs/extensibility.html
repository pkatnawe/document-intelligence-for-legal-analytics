Part 1 · Architecture

Extensibility & Scale

The brief asks for two forward-looking things: room to add LLM summarization/Q&A later, and a story for scaling to large volumes. Both come down to choices made now.

TL;DR
  • Data is stored so summarization/Q&A is a later add-on, not a rewrite (raw + page-chunks + embeddings + metadata).
  • Agentic flows follow for free: the agent's tools (SQL, vector search, cross-doc joins) are the artifacts the pipeline already produces, governed by Unity Catalog.
  • Scales gracefully: async + Workflows + model tiering, moving to provisioned throughput at high volume.
  • User-facing latency stays low — the API returns a job id in milliseconds; extraction runs off the request path.

Designing today so summarization / Q&A is cheap tomorrow

Q&A over documents is retrieval-augmented generation: find the relevant passages, then ask an LLM over them. We don't build it now — but a few storage choices today make it a small addition instead of a rewrite: keep the raw documents (not just extracted fields), chunk text by page so answers can cite an exact page, embed those chunks into a vector index, and tag every chunk with client, date, and party so retrieval can be filtered. Unity Catalog keeps it all access-controlled.

Everything left of the dotted box we build during the POC anyway. The Q&A feature on the right just plugs into the searchable index — in our stack, simply another DSPy signature reusing the same models. Citations work because we split documents by page, so every answer can point back to its source.

From Q&A to agentic flows

Single-shot RAG answers one passage-level question. The Claims questions that matter are multi-step and cross-document — "summarise everything about claim #1234: what was billed, what the contract obliges, what was agreed in the minutes, and what's still open." That's an agent: it plans, calls tools, reasons over the results, and cites its sources. The key point for the brief's "extensibility" ask is that we are not adding new infrastructure for this — the agent's tools are the artifacts the POC already produces.

So the progression is deliberate: extract → store well → retrieve → answer → act. Each step is the substrate for the next, which is why summarization, Q&A, and eventually agentic Claims workflows are configuration over the same data, not a second system.

The hardest queries — and why the design already handles them

A good slide names the hard cases. The pattern: the genuinely hard queries are structured ones, and those are cheaper and more reliable without an LLM — reserve the LLM for the semantic tail.

Hard queryWhat handles it
"Total spend for client X in 2024" (aggregation)A SQL query over Gold tables — no LLM
"All docs for Acme" when it's also "Acme Corp." (entity resolution)A normalization step in Silver → one canonical client id
"Invoices tied to the contract mentioning clause Y" (linking)Shared metadata keys (client, claim id) across doc types
"Where do we agree to indemnify?" (semantic)Vector search over page chunks → cited passages
"What changed about this claim over time?" (temporal)Delta time-travel + the audit log

Scaling as the volume grows

Documents arrive two ways — single files through the API, and batch drops from cloud storage, email, or scanner output — and both feed the same queue, so scale is one problem, not two. The design grows gracefully: the same codebase, with a lever pulled at each stage rather than a rewrite.

As volume grows…What changes (same code)
POC — hundreds–thousandsPay-per-token model endpoints; background worker inside the App
Growing — tens of thousands/dayHeavy work moves to Databricks Workflows (autoscaling workers) behind a durable queue; Delta partitioned by date/client
High volumeModel endpoints switch from pay-per-token to provisioned throughput — guaranteed capacity that autoscales, since pay-per-token has per-minute rate limits and isn't built for high throughput[3]

Two things hold across all stages: cost stays controlled by the model cascade (cheap model on the bulk, premium only on the hard tail — up to ~98% cheaper than premium-everywhere[1]), and reliability comes from idempotent jobs, retries with backoff, and a dead-letter path so one bad document never blocks a batch.

Real-world latency

Task 3 asks for real-world latency, and the core answer is structural: the API is async, so a POST returns a job id in milliseconds — the slow LLM work never sits on the request path, and user-facing latency is independent of how long extraction takes.

Processing itself stays fast for concrete reasons: model tiering means most docs return from a low-latency model and only the hard tail pays the premium cost; we send only the relevant pages, not whole documents; embeddings are precomputed at ingest so future search is a quick lookup; identical resubmissions are cached. SLA framing: the API accepts-and-returns immediately (HTTP 202) and extracts on a worker, so the API's response time is decoupled from extraction time — the standard async-accepted pattern. Extraction time itself is dominated by the tier-1 model for the majority of documents, with tier-2 only on the escalated tail.

The model tiers. The deployed POC runs the open-weight tiers Free Edition serves: tier-1 gemma-3-12b (vision — reads the scans) → tier-2 meta-llama-3-3-70b-instruct (text). Production swaps to tier-1 gemini-3-5-flash (fast, vision-capable; Google global endpoint, so cross-geo routing must be enabled) → tier-2 claude-opus-4-8 ("complex extraction… document understanding," vision-capable, hosted inside the Databricks perimeter).[2] With DSPy, switching tiers is a one-line change.
