Technical Debt & Scaling

What's weakest in this build, named before the panel names it — and the concrete things that have to change as volume grows. A POC earns trust by being honest about its edges, not by hiding them.

The single weakest point: on Free Edition the open-weight vision model (gemma-3-12b) misreads dense receipts — it got the Uber total wrong. We don't hide that: the reconciliation check catches it and flags the job for human review rather than storing a confident wrong answer, and production swaps tier-2 to the vision-capable claude-opus-4-8, which reads it correctly. The accuracy gap is a model-tier choice (config), not a pipeline flaw — but it's the first thing I'd put on the table.

Key things to consider as we scale

The architecture is designed so each of these is a lever pulled at one stage, not a rewrite — but each needs a real decision as volume grows.

⚙️ Throughput & workers

In-app worker → durable queue + Databricks Workflows autoscaling. Make jobs idempotent (job id is the key), add retries + a dead-letter path so one bad document never blocks a batch, and a sweeper to recover orphaned RUNNING jobs after a restart.

🧠 Model serving

Pay-per-token has per-minute rate limits → move to provisioned throughput for guaranteed, autoscaling capacity. Keep the cascade (cheap bulk, premium tail) as the primary cost lever; consider request batching and only sending the relevant page(s).

🗄️ Storage & concurrency

Split the hot path (job status → transactional store) from the cold path (results + append-only audit → Delta). Partition Delta by date/client; manage the UC Volume's raw-PDF lifecycle (retention/archival). Watch Delta optimistic-concurrency conflicts on status writes.

🔐 Governance & multi-tenancy

Unity Catalog isolates one client's documents from another's. Add PII redaction before model calls and pin model placement (in-perimeter vs cross-geo) for sensitive data — the confidentiality work we flagged but didn't build.

📊 Observability at volume

MLflow autolog traces every call; at scale, store traces in Unity Catalog (past the 100k-trace limit) and alert on escalation-rate and reconcile-flag-rate spikes (the early signal of drift or a bad vendor template).

📈 Quality & drift

Grow the gold set from human-flagged cases; re-score every release in MLflow; GEPA-optimize the prompt and tune thresholds on the firm's real invoices. Improvement becomes measured, not anecdotal (see ✓ Evaluation).

What does NOT need to change to scale

Worth saying out loud — the load-bearing choices hold up:

  • The async contract (202 + poll) already decouples user-facing latency from extraction time — the API doesn't get slower as the model does.
  • Each document is independent, so extraction is embarrassingly parallel — scaling is adding workers, not redesigning.
  • The cascade keeps cost flat-ish as volume grows (the cheap model absorbs the bulk).
  • The JobStore interface is the seam: SQLite-style → Delta → transactional, no change to the service logic.
  • Model choice is config, not code — swap tiers (or whole providers) with env vars.
The framing: none of this is hidden risk — it's a scoped POC with a clear upgrade path at every stage. The design pays for itself by making each scale lever a config or interface change, and by never letting a known weakness (a weak model) turn into a silent wrong answer.
