Platform Architecture

The high-level slide: one picture of the whole system, with the three things the brief asks about — failure handling, scale, and audit — marked on it.

TL;DR
  • One pipeline serves all document types; uncertain cases escalate to a stronger model, then to a human.
  • The cheap path handles ~90% of documents; the premium model runs only on the hard tail.
  • Failure handling, horizontal scale, and a Databricks-native audit trail are built into the design.

Seven numbered stages: ingest → keep the original → classify → extract → check → store → use. The key detail is the middle: classify uses a cheap rules scorer and only escalates to an LLM when it's unsure, and from there each document type is routed to its own extractor and schema (invoice, contract, email/minutes) — that's how we handle the firm's mix of formats. Anything uncertain (unknown type, or output that fails the check) goes to human review, never guessed. The purple lane is the audit trail every 📝 step writes to — mostly Databricks-native (Delta history, Unity Catalog lineage, MLflow), so any document's full history reconstructs without us building a logging system. Detail in ⑤.

All input types — and the image-only case, handled explicitly: PDF, Word, Excel, and images all enter the same pipeline. The split that matters is text layer or not — digital files are read directly; image-only scans and photos are rendered to a page image, OCR'd for classification, and read by the vision model for extraction. "Documents that are just pictures" are a first-class path, not an afterthought.

The three things the slide has to answer

⚠ Failure

Every stage can fail on its own. Anything the system isn't sure about — a doc it can't classify, output that won't validate — is quarantined, not dropped or guessed. Nothing fails silently.

📈 Scale

Ingest is decoupled from extraction, so workers fan out horizontally. Each document is independent, so they can be processed in parallel across many workers.

🧾 Audit

An append-only log records every state change; Delta time-travel records every data change. Together they answer "what happened, and what did the data look like then." Detail in ⑤.

⚡ And it's fast: the API is async (a job id comes back in milliseconds), a cheap model handles the majority while only the hard tail escalates to a premium one, and embeddings are precomputed at ingest. Fast because the slow work is off the request path; accurate because hard docs still reach the premium model — full reasoning in ⑥.

A note on the extract stage

The case names the facts to pull — clients, amounts, dates, parties — so each document type has its own target schema. We express that as a DSPy typed signature: declared inputs, a Pydantic output schema. That single object is both the model interface and the validation contract, which is exactly the "separate LLM interaction from validation" the brief asks for — adding a new document type just means adding a new signature. (More on that idea in ★ Concepts.) Classification stays a separate step from extraction — a decoupling shown to raise accuracy with minimal supervision.[1]
