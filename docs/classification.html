Part 1 · Architecture

Classification

Before the system can extract anything, it has to know what it is looking at — invoice, contract, email, minutes. This page covers how documents are sorted, and why that step is intentionally simple.

TL;DR
  • Documents are sorted by a deterministic rules scorer — explainable, auditable, and effectively $0 per document.
  • Only the ambiguous tail escalates to an LLM; each document type routes to its own extractor and schema.
  • Accuracy improves offline (via GEPA) without sacrificing the deterministic, auditable inference.

Rules, not an LLM — and the reason is legal, not cost

The reflex is to ask an LLM "what kind of document is this?" For a legal client, that's the wrong first move. Picture having to explain to an auditor why a document was classified as an invoice. With an LLM, the honest answer is "the model thought so." With a scoring rule, it's concrete: "these words appeared, here's what each is worth, it cleared the bar." One answer survives scrutiny; the other doesn't. So we classify with plain rules over the text — deterministic, explainable, and auditable.

How the scorer thinks

Roughly how you'd eyeball a page in two seconds — three signals: the words ("invoice number," "bill to," "amount due" — weighted, since "invoice number" is far stronger evidence than "subtotal"), the shape (an invoice-number pattern, currency totals, a line-item table — which tells real invoices apart from contracts that merely mention one), and the envelope (file type and length: emails are .eml, contracts are long, minutes have "action items"). Highest score wins — but only if it clears a confidence threshold. If nothing's convincing, the doc is marked unknown rather than force-fit.

The threshold doubles as an input guard. Because the POC handles invoices, the same confidence check answers "is this actually an invoice?": a contract sent to the invoice extractor is rejected up front — with a reason and the signals that fired — rather than having invoice fields hallucinated onto it. This adds failure-handling at no extra cost.

Where it goes next: a ladder, not a dead end

Rules are a great start, not a forever home. The honest roadmap is a ladder you climb only when forced: rules → classic ML (TF-IDF + logistic regression, once you have labels) → embeddings (when wording drifts) → an LLM (only for the ambiguous tail). What ties the rungs together is confidence-gated escalation — the cheap path handles what it's sure about, and only the uncertain remainder escalates.

flowchart LR D(["📄 Document"]) --> R["📋 Score with simple rules"] R --> T{"Confident? 📝"} T -->|"yes ~90%"| OUT["✅ Tag it → route to that type's extractor
(record which words fired) 📝"] T -->|no| ESC["🧠 Ask the LLM"] ESC --> H[("📥 Keep the hard case
+ the human's fix")] H -->|"becomes"| TRAIN["🎓 Training data"]

Down the "yes" path is the easy ~90% — tagged instantly and 📝 recorded with the exact words that fired (that's the auditable part). The rest escalates to the LLM, and those hard cases plus any human fix become training data. Caveat: the useful labels are the hard cases — training on the easy ones the rules already handle just teaches a model to copy the rules.

Making it better automatically — "optimize anything" (GEPA)

The classifier is just keyword weights and thresholds in a config file — text with a measurable score (its accuracy on the gold set). Anything that's text-with-a-score can be tuned by GEPA, a reflective optimizer that reads the failures, proposes an improved version, and keeps a frontier of the best variants — beating reinforcement learning by ~6–19 points while using up to 35× fewer trials.[3]

So we point GEPA at the misclassified examples the eval loop collects, and it tunes the weights and thresholds for us. Crucially the optimization runs offline — at inference the classifier is still the same fast, deterministic, explainable rules. We get learned-quality accuracy without giving up the auditability that made rules the right call. (The same tool, as dspy.GEPA, can later optimize the extraction prompts too.)

Documents that are just pictures — classification-guided VLM, not OCR

A scan has no text layer. The obvious option is OCR — transcribe the image to characters, then treat it like text. We rejected OCR as the reading mechanism for three reasons that compound at scale: it flattens spatial layout (an invoice's meaning lives in its geometry — line-item tables, a total positioned bottom-right, columns), it's lossy on degraded scans (seals, low contrast, skew), and a single OCR pipeline doesn't generalize across the firm's diverse layouts — every new template is new tuning.

Instead we use a classification-guided vision-language model: a lightweight classifier (the Nature paper's training-free feature matching, or a small trained image model) predicts the document type from the page image, that type selects a type-specific prompt + schema, and the VLM reads the image directly — layout intact. Decoupling classify-from-extract this way beat a strong baseline by ~18 F1 points zero-shot, and removing the classify step alone cost 18 F1 — it's the dominant accuracy lever.[2]

The known risk of a VLM is the opposite of OCR's: under degradation it can hallucinate a plausible-but-wrong value rather than signal it can't read.[1] We don't avoid the VLM to dodge that — we catch it downstream: schema validation, the line-items-vs-total reconciliation, and human-flagging mean a hallucinated value is surfaced, never silently stored. And the classifier + the type-specific prompts are optimized later with GEPA on the firm's own documents.

Where this could be wrong: classification-guided gains scale with a clean, finite set of document types — if the intake taxonomy is noisy, a misclassification routes the wrong prompt and propagates structured errors. Mitigated by the confidence gate (unsure → human) and by tuning on a sample of the firm's actual documents.
Scope: all of this is Part-1 platform thinking. The Part-2 invoice code skips classification entirely (invoices only) — none of this classifier machinery ships; it reads the page image with the VLM directly. See the code-vs-slides boundary.
